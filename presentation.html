Foundation — Step 1 of 3

It Started as Next-Token Prediction

The model predicts the most likely next token, then repeats. No planning, no memory — just statistics over text.

Prompt: "def is_even(n):\n return "
Model predicts next token:
n % 2 == 0 ← 71%
n % 2 ← 18%
not n % 2 ← 8%
Then repeats for the next token, and the next…
Limitation: no reasoning, no planning. Ask "is 17 prime?" and it pattern-matches its way to an answer — right or wrong. Good for autocomplete; not enough for complex tasks.
Foundation — Step 2 of 3

Add Thinking: Reason Before Answering

Chain-of-thought prompting (and later o1-style reasoning) lets the model plan before committing to an answer. Huge jump in capability.

Task: "This test fails intermittently. Why?"
<thinking>
The test touches a shared cache...
Two threads could write simultaneously...
Race condition on cache.set() — not thread-safe...
Fix: add a lock, or use an atomic operation.
</thinking>
"The test has a race condition on the shared cache. Add a threading.Lock around cache.set()."
What changed: the model explores multiple hypotheses silently before answering. Longer tasks, fewer confident wrong answers. This is where models got genuinely useful for engineering work.
Foundation — Step 3 of 3

Add Tool Calls: Act in the World

The model can now pause, call a tool, observe the result, and continue reasoning. It's no longer just text in, text out.

Task: "Run the tests and fix any failures."
→ tool: run_shell("pytest tests/")
FAILED test_user.py::test_login — AssertionError: 401
→ tool: read_file("src/auth.py")
…token expiry check has an off-by-one on seconds…
→ tool: edit_file("src/auth.py", fix)
→ tool: run_shell("pytest tests/") ✓ all passed
= Agent: think → act → observe → repeat. This loop is what makes it possible to hand off a 30-minute task and come back to a PR.
Part 1 — The New Reality

The Models Finally Got Good

We've moved from autocomplete to reasoning agents. The best way to measure that? Feedback loop length — how long an agent can work on its own before it needs you.

  • Longer feedback loop = more complex tasks the agent can handle
  • We went from autocompleting a line to 30-minute unsupervised tasks
Key shift: agents handle execution now. Humans provide direction. But we still read every line — we just don't write boilerplate anymore.

Part 2 — The Lifecycle

Planning Before Code

Agree on what to build before any code is written. OpenSpec is a lightweight framework that structures this as a folder of Markdown artifacts.

The workflow

/opsx:propose dark-mode
→ openspec/changes/dark-mode/
proposal.md ← why & what
specs/ ← requirements
design.md ← approach
tasks.md ← checklist
You review and refine each artifact
with the agent until it's locked.
/opsx:apply → agent implements all tasks
/opsx:archive → spec merged into codebase

specs/dark-mode-spec.md

### Requirement: Theme toggle
- SHALL render a toggle in the top-right header
- SHALL apply the new theme within 50ms
- SHALL persist choice to localStorage
#### Scenario: Enable dark mode
- GIVEN the user is on any page
- WHEN they click the dark mode toggle
- THEN the theme switches within 50ms
- AND the preference is saved
#### Scenario: Page reload
- GIVEN dark mode was previously set
- WHEN the page reloads
- THEN dark mode is applied before first render
Why this works: the spec is the contract between you and the agent. Changing a requirement in Markdown costs seconds. Changing it after 200 lines of generated code costs hours.

Part 2 — The Lifecycle

The Validation Bottleneck

Coding speed is no longer the bottleneck. Reviewing is. The bottleneck just moved.

  • We can produce code faster than we can validate it
  • CodeRabbit on every PR — automated sanity checks, style, logic issues
  • But automated review isn't enough — humans are still the quality gate
  • The review burden is now the constraint on velocity
Reality: you're not "writing less." You're reviewing more. The skill is shifting from syntax to judgment.
Part 2 — The Lifecycle

Agentic Exploratory Testing

To speed up validation, we're using agents for exploratory testing — beyond the traditional testing pyramid.

  • Agent reads the PR diff and documentation
  • Agent reasons about which UI components are affected
  • Agent scripts Playwright on the fly to test those paths
  • Agent "looks" at screenshots to verify behavior
Key difference: not a fixed test suite. It's a flexible reasoner exploring what changed. Blurs the line between testing and synthetics monitoring.
Part 2 — The Lifecycle

Dev Meets Ops

The loop closes itself. A production error becomes a GitHub Issue becomes a PR — with minimal human initiation.

1. Discover: ES|QL CATEGORIZE clusters millions of logs → ranked error patterns
2. Investigate: Triage Agent runs a protocol-driven investigation — queries telemetry, checks git history, correlates stack traces
3. File: Agent creates a GitHub Issue with root cause, affected versions, and a suggested fix
4. Fix: Coding Agent picks up the issue, drafts a PR
5. Validate: CodeRabbit + Exploratory Tests run on the PR
6. Ship: Human approves → merged → deployed
Real example (elastic.co): a sporadic TypeError in Kibana's Streams UI was found, root-caused to a race condition, and reported with a fix suggestion — fully autonomously. The engineer only entered the loop to review the PR. Read the full write-up →
Part 3 — The Stack

You Are the Orchestration Layer

Each agent runs its own think→act→observe loop independently. You switch between them — assigning tasks, reviewing outputs, steering direction.

How it works today

  • OpenCode runs in each TMux pane — one agent per task/branch
  • TMux is the session manager: persistent, switchable, parallel
  • Assign Agent A a 30-min task → switch pane → assign Agent B → come back, review, correct → repeat
  • You're the scheduler; agents are the workers

The next level

  • Gastown (by Steve Yegge) — agents orchestrate agents: a "Mayor" agent creates convoys, assigns tasks to worker agents, tracks work in git-backed beads
  • You describe the goal at a high level; Gastown breaks it into a multi-agent effort and coordinates execution
  • Still early — but the direction is clear: the human moves one more level up from execution
Mental model: TMux is your cockpit. Each pane is an agent in flight. You're air traffic control — not the pilot.
Part 3 — The Stack

How You Actually Pay for Models

Forget subscriptions — API pricing is per token. A token is roughly ¾ of a word. You pay separately for what you send in and what the model generates.

The two meters

Input tokens — everything you send:
system prompt + conversation history
+ files, specs, context you inject
Output tokens — what the model writes:
thinking tokens (if reasoning model)
+ actual response / generated code
⚠ Output tokens cost 3–5× more than input.
Long reasoning chains get expensive fast.

Example: one coding task

10 000 input tokens (context + spec)
+ 2 000 output tokens (code + explanation)
Opus 4.6: $0.05 + $0.05 = $0.10
GLM 5.1: $0.0095 + $0.0063 = ~$0.016
Step Flash: $0.001 + $0.0006 = ~$0.002
Subscription plans flatten this — but they're
subsidized and won't stay cheap. Budget per-token.

  • Architecture & planning
  • High-level reasoning

GLM 5.1

$0.95 in / $3.15 out per 1M tokens
  • Open-source (Z.ai)
  • Coding & triage
  • More than capable for the actual work

Step 3.5 Flash

$0.10 in / $0.30 out per 1M tokens
  • ~50× cheaper than Opus on output
  • MoE, 196B params, 11B active
  • Great for high-volume loops
Strategy: use Opus for architecture and planning, GLM or Step Flash for the actual coding. The cost difference is an order of magnitude — and you won't notice the quality difference on most tasks.
Part 3 — The Stack

Context Is the New Code

The model is only as good as what you feed it. The real work is curating context.

  • Markdown "Skills" — reusable capabilities. Example: analyze_user_feedback.md knows how to summarize thousands of support tickets
  • Shared repos with helper scripts that fetch from BigQuery, telemetry, web search — all into the context window
  • PMs use Cursor + agents for competitive analysis, feature prioritization — not just engineers anymore
Pattern: define the skills, pull in the data, agent connects the dots. Context engineering is the new software engineering.
Part 4 — The Human Toll

The Death of "The Zone"

This workflow changes how your brain works. It's not all upside.

  • No more 4-hour deep dives — constant context switching
  • Pattern: assign Agent A a 30-min task → switch → assign Agent B → review A → correct → loop
  • More code review, less flow state
  • As a tech lead, it lets me get closer to the code — but some engineers will struggle with the shift
Mental model: your brain becomes a CPU scheduler. Different skill, different feel. Be mindful of the transition.
Part 4 — The Human Toll

Workflow Gentrification

When it's easy to build, it's easy to build things you don't need. This is a real problem.

  • Because it's easy to build, we build features that don't solve real problems
  • AI-written code still needs human-led maintenance — you still own the debt
  • 20% more productivity can mean 40% more technical debt if you're not careful
  • Code sprawl is real — more surface area for bugs, more to understand
Warning: the constraint isn't "can we build it." It's "should we build it." Discipline matters more than ever.

Part 5 — Conclusion

Code Is the Universal Language

Agents work best when you turn problems into code. This isn't making code less important — it's making it more important.

  • Code is what agents speak fluently. Turn everything into code — config, specs, infra, tests
  • More people will code (at least a little) because the barrier to entry dropped
  • Jevons paradox: making development cheaper increases demand for software — and therefore for more engineers, not fewer
  • We need people who understand systems: judgment, architecture, context — that doesn't automate away
Bottom line: the engineer's job shifts from writing to directing. Agents handle execution — you handle judgment, architecture, and context. The skill changes; the need for engineers doesn't.
